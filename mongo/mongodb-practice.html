⏱ Problem 1 — ~5 minutes

Problem 1 — Design Your Database

The Scenario

Sunrise Café is a small campus coffee shop. They want to move their record-keeping into MongoDB. Here is exactly what they need to store.

  • Every item on the menu has a name, a category (either "drink" or "food"), a price in dollars, and a flag that says whether the item is currently available for ordering.
  • Every order placed by a customer records the customer's name, the name of the item they ordered, how many of that item they took, the total dollar amount charged for the order, and whether the customer has already paid.

Answer these three questions in your notes

  1. What will you name the database?
  2. How many collections do you need, and what will you name each one?
  3. For each collection, list every field name and its type — String, Number, or Boolean.
Hint: there should be exactly two collections. Ask yourself: should the menu know anything about orders, or should they be stored separately? Keep the two concerns apart.

Problem 3 — Write Five Queries

Run in: mongosh

Write one find() command for each task below. The expected result is described under each one — your output must match it exactly.

Q1 Find all menu items in the "drink" category.

Expected: 3 documents — Espresso, Latte, Green Tea.

Q2 Find menu items where the price is strictly greater than $3.00.

Expected: 3 documents — Latte ($4.00), Bagel ($3.50), Croissant ($3.25). Green Tea ($3.00) should not appear.

Q3 Find menu items that are food and are currently available.

Expected: 2 documents — Muffin, Bagel. Croissant is food but unavailable, so it must not appear.

Q4 Find orders where the total is $7.00 or more. Show only customer and total — hide _id. Sort by total from highest to lowest.

Expected — in this exact order:
{ customer: "Carol", total: 8.25 }
{ customer: "Alice", total: 8.00 }
{ customer: "David", total: 7.00 }

Q5 Find orders placed by Alice or Eve.

Expected: 2 documents — Alice's order and Eve's order. No other customers should appear.
Operators needed across these five queries: a comparison operator for Q2  ·  two conditions matched at once for Q3  ·  projection and sort chained together for Q4  ·  match against a list of values for Q5.

Any sensible name works — cafe, sunrise, coffeeshop. What matters is that you use it consistently for the rest of the problems.

✓ ANSWER KEY — Problem 3: Queries Q1–Q3

Answer 3A — Queries

Run in: mongosh

Q1 All drinks

▷ COMMAND
> db.menu.find({ category: "drink" })
Returns: Espresso, Latte, Green Tea

A plain field-value match — no operator needed.

Q2 Price strictly greater than $3

▷ COMMAND
> db.menu.find({ price: { $gt: 3 } })
Returns: Latte (4.00), Bagel (3.50), Croissant (3.25). Green Tea (3.00) is excluded — $gt means strictly greater, not equal.

Q3 Food items that are currently available

▷ COMMAND
> db.menu.find({ category: "food", available: true })
Returns: Muffin, Bagel. Croissant is food but has available: false, so it is excluded.

Two fields in one filter object is an implicit AND. You do not need $and here — it is the shorter idiomatic form.

If you used $and for Q3, it also works:
db.menu.find({ $and: [{ category: "food" }, { available: true }] })
Both forms produce identical results. Prefer the shorter form when matching plain values on the same document.
✓ ANSWER KEY — Problem 3: Queries Q4–Q5

Answer 3B — Queries

Run in: mongosh

Q4 Orders ≥ $7 · projection · sort descending

▷ COMMAND
> db.orders.find(
{ total: { $gte: 7 } },
{ customer: 1, total: 1, _id: 0 }
).sort({ total: -1 })
Returns in this order:
{ customer: "Carol", total: 8.25 }
{ customer: "Alice", total: 8.00 }
{ customer: "David", total: 7.00 }

$gte: 7 includes 7.00. The second argument to find() is the projection — _id: 0 explicitly hides the id field. sort({ total: -1 }) means highest first.

Q5 Orders placed by Alice or Eve

▷ COMMAND
> db.orders.find({ customer: { $in: ["Alice", "Eve"] } })
Returns: Alice's order and Eve's order — 2 documents.

$in matches any document where the field value appears anywhere in the given array.

Alternative that also works:
db.orders.find({ $or: [{ customer: "Alice" }, { customer: "Eve" }] })
Both are correct. Use $in when you are checking the same field against multiple values — it is shorter and easier to read.
✓ ANSWER KEY — Problem 4: Update + Delete

Answer 4 — Update and Delete

Run in: mongosh

Updates

▷ U1 — RAISE ESPRESSO PRICE
> db.menu.updateOne(
{ name: "Espresso" },
{ $set: { price: 3.00 } }
)
▷ U2 — RESTOCK ALL UNAVAILABLE ITEMS
> db.menu.updateMany(
{ available: false },
{ $set: { available: true } }
)
# Matches Green Tea and Croissant — both become available: true
▷ U3 — MARK BOB AS PAID
> db.orders.updateOne(
{ customer: "Bob" },
{ $set: { paid: true } }
)

Deletes

▷ D1 — REMOVE CAROL'S ORDER
> db.orders.deleteOne({ customer: "Carol" })
# 4 orders remain: Alice, Bob, David, Eve
▷ D2 — REMOVE ALL UNPAID ORDERS
> db.orders.deleteMany({ paid: false })
# At this point: Bob is paid (U3), Carol is gone (D1)
# Only David is still unpaid — he is removed
# 3 orders remain: Alice, Bob, Eve
Key insight for D2: running commands in sequence changes the state of the collection. U3 made Bob paid before D2 ran, so Bob survives the delete. Students who said only 2 documents remain likely forgot U3.
