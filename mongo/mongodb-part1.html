What Is a Database?

  • A database is a system that stores data outside your application code.
  • It lets your app save data, find data later, and update it safely.
  • It keeps data available even after the server restarts.
  • It gives you tools for search, filter, sort, and aggregation.
Simple idea: your Express app handles requests; the database handles long-term storage.

If We Do Not Use a Database

Typical classroom demo

▷ JavaScript memory only
let tasks = [
{ id: 1, title: "Buy milk" },
{ id: 2, title: "Finish assignment" }
];

Problems

  • Server restarts → array disappears
  • Crash → data disappears
  • Multiple servers cannot reliably share the same array
  • No built-in persistence or indexing
Bottom line: in-memory data is fine for a tiny demo, not for a real application.

If We Use a Database

What improves

  • Data survives restarts
  • Many requests can read/write safely
  • Queries become fast and expressive
  • Different app servers can use the same data store

Two Big Database Families

Think of databases like different ways of organizing information. One is like neat spreadsheets. The other is like storing complete info cards.

SQL Like a spreadsheet with strict columns

  • Data is stored in tables, like Excel sheets
  • Each row is one item, like one student
  • Each column has a fixed meaning, like name, major, age
  • Every row should follow the same structure
  • Different tables can connect to each other, like using student IDs
Example table: students

| id | name  | major | age |
|----|-------|-------|-----|
| 1  | Ava   | HCDD  | 20  |
| 2  | Omar  | IST   | 21  |
Good when data is very organized and predictable. Examples: PostgreSQL, MySQL, SQLite

NoSQL Like storing full profile cards

  • Data is often stored as documents
  • Each document can hold all details for one item in one place
  • Not every document must look exactly the same
  • This is helpful when data has nested details or changes often
  • MongoDB stores data this way
Example document: one student
{
  "name": "Ava",
  "major": "HCDD",
  "age": 20,
  "courses": ["HCDD 340", "IST 256"],
  "address": {
    "city": "State College",
    "state": "PA"
  }
}
Good when data is more flexible or has nested information. Examples: MongoDB, Redis, Cassandra
Simple way to remember:
SQL = “Put everything into neat tables.”
NoSQL = “Store each item as a complete flexible record.”

SQL Example: Student Data in Tables

Tables

students

102BobIST
103RinaCS
enrollments
student_idcourse
101HCDD 411
101IST 220
102IST 210
103CMPSC 132
Notice the layout: columns are field names, and rows are records. That is the normal relational-table shape.

Run in: SQL database client

▷ Example 1: Find Alice's courses
SELECT s.name, e.course
FROM students s
JOIN enrollments e ON s.id = e.student_id
WHERE s.id = 101;
▷ Example 2: Find all HCDD students
SELECT id, name, major
FROM students
WHERE major = 'HCDD';
SQL is excellent when data has stable structure and you need joins, filters, and shared tables often.

NoSQL Example: Same Student Data in MongoDB

students collection

▷ Same data as slide 7, modeled as documents
{ "studentId": 101, "name": "Alice", "major": "HCDD", "courses": ["HCDD 411", "IST 220"] }
{ "studentId": 102, "name": "Bob", "major": "IST", "courses": ["IST 210"] }
{ "studentId": 103, "name": "Rina", "major": "CS", "courses": ["CMPSC 132"] }
This is the same underlying student data from slide 7. The difference is only the data model: MongoDB keeps each student's related course data together inside one document.

Run in: mongosh

▷ Example 1: Find Alice's courses
> db.students.findOne(
... { studentId: 101 },
... { _id: 0, name: 1, courses: 1 }
... )
▷ Example 2: Find all HCDD students
> db.students.find(
... { major: "HCDD" },
... { _id: 0, studentId: 1, name: 1, major: 1 }
... )
MongoDB is convenient when the data is naturally JSON-like and you often want related values, like a student's courses, in the same record.

When to Choose SQL

  • Banking, payments, accounting, ledgers
  • Inventory systems with strict constraints
  • Applications with many joins across related tables
  • When the schema is known and changes slowly
  • When strong transactional guarantees are central to the system
Typical classroom shortcut: “structured tables + stable relationships + lots of joins” usually points toward SQL.

When to Choose NoSQL

  • User profiles with optional fields
  • Product catalogs where products differ from one another
  • CMS content, blog posts, comments, metadata
  • Rapid prototyping where the schema changes often
  • Data that already looks like JSON objects
Typical classroom shortcut: “flexible JSON-like documents + nested data + evolving schema” often points toward MongoDB.

JavaScript end-to-end

Good tooling

  • Compass for visual exploration

Same data, three stages

The key idea is that the shape of the data stays almost the same. Only the environment changes: browser request, Express app, then MongoDB storage.

1) Client JSON

▷ Sent by browser / Postman
{
"name": "Alice",
"major": "HCDD",
"year": 2
}

2) Express Object

▷ In req.body
{
name: "Alice",
major: "HCDD",
year: 2
}

3) BSON Document

▷ Stored in MongoDB
{
"_id": ObjectId("..."),
"name": "Alice",
"major": "HCDD",
"year": 2
}
Takeaway: frontend sends JSON → Express reads a JavaScript object → MongoDB stores nearly the same data as a BSON document, often with an added _id.

What the Same Student Data Looks Like as One MongoDB Document

▷ One document from the students collection
{
"_id": ObjectId("67e41fc..."),
"studentId": 101,
"name": "Alice",
"major": "HCDD",
"courses": ["HCDD 411", "IST 220"]
}
This is the MongoDB version of the Alice rows from slide 7: one row from students plus Alice's matching rows from enrollments, combined into one document.

Notice

  • _id is automatically added if you do not provide one
  • studentId, name, and major match the SQL student row
  • courses holds what SQL stored in separate enrollment rows

Why this matters

  • You can read one document instead of joining two tables
  • The data stays close to the JSON shape your API may already use

Embedding vs Referencing

Embedding

▷ One document contains related data
{
"title": "1984",
"author": { "name": "George Orwell", "born": 1903 }
}
  • Good when data is read together
  • Good when nested data is small

Referencing

▷ Separate documents linked by id
{ "title": "1984", "author_id": ObjectId("1111") }
{ "_id": ObjectId("1111"), "name": "George Orwell" }
  • Good when related data is shared widely
  • Good when related items change independently

Local Setup: What You Need to Install

1. MongoDB Community Server

  • This is the actual database engine
  • Without this, there is no local MongoDB server running

2. mongosh

  • This is the command-line shell
  • You type database commands here

3. Compass

  • This is the GUI application
  • Optional but strongly recommended for beginners
Common beginner mistake: Compass is not the database server. It is only a visual interface for connecting to a server.

Download Strategy Before You Install

  1. Install MongoDB Community Server first.
  2. Install mongosh second.
  3. Install Compass third.
  4. Start the server.
  5. Connect with mongosh or Compass.

macOS path

  • Homebrew is the easiest route
  • Use Terminal for installation and service commands

Windows path

  • MSI installer is easiest
  • Use File Explorer + installer wizard + Command Prompt/PowerShell

macOS: Full MongoDB Setup, Step by Step

Goal: install the MongoDB server, start it, install mongosh, and connect to your local database on a Mac.

Step 1 — Check the prerequisites

  • You need the Terminal app
  • You need Xcode Command Line Tools
  • You need Homebrew
Run in: macOS Terminal
▶ CHECK / INSTALL PREREQUISITES
xcode-select --install
brew --version
If brew --version fails, install Homebrew from the official Homebrew website, then come back and rerun the command.

Step 2 — Install MongoDB Community Server

Run in: macOS Terminal
▶ INSTALL SERVER
brew tap mongodb/brew
brew update
brew install mongodb-community@8.0
  • brew tap ... adds MongoDB's package source to Homebrew
  • brew update refreshes Homebrew's package list
  • brew install ... installs the MongoDB server on your Mac
Do not run these inside: mongosh or Compass. These are Terminal commands.

Step 3 — Start MongoDB and verify that it is running

Run in: macOS Terminal
▶ START / CHECK / STOP
brew services start mongodb-community@8.0
brew services list
brew services stop mongodb-community@8.0
  • Use start to launch MongoDB as a background service
  • Use list to confirm the service shows as started
  • Use stop only when you want to shut the local server down
Best beginner path: use brew services start .... It is simpler than manually starting mongod.

Step 4 — Optional manual start command

Run in: macOS Terminal
▶ OPTIONAL MANUAL START
mongod --config /opt/homebrew/etc/mongod.conf --fork
On Intel Macs, the config path may be /usr/local/etc/mongod.conf instead of /opt/homebrew/etc/mongod.conf.
Use this only if you are intentionally starting MongoDB manually or troubleshooting. Most students should stay with brew services.

Step 5 — Install the MongoDB shell

Run in: macOS Terminal
▶ INSTALL MONGOSH
brew install mongosh
mongosh --version
This installs the shell program you will use to type MongoDB commands interactively.

Step 6 — Connect to your local MongoDB server

Run in: macOS Terminal
▶ CONNECT
mongosh
  • If MongoDB is running locally, this opens the MongoDB shell
  • Now you can type MongoDB commands like show dbs and use school

Windows: Full MongoDB Setup, Step by Step

Goal: install the MongoDB server on Windows, set it up the easy way as a service, install mongosh, and connect to your local database.

Step 1 — Download the MongoDB Community Server installer

  • Go to the MongoDB Community Server download page
  • Choose Windows as the platform
  • Choose the MSI package
  • Download the current course version or current stable version
At this point, you are using the browser and File Explorer, not Terminal or mongosh.

Step 2 — Run the installer and pick the beginner-friendly options

Do this in: Installer wizard
  1. Double-click the downloaded .msi file
  2. Choose Complete setup unless your instructor says otherwise
  3. Keep Install MongoD as a Service checked
  4. Use the default Network Service user
  5. Let the installer create the data and log directories
  6. Keep Install MongoDB Compass checked if it is offered
  7. Click Install and finish the wizard
Best beginner choice: installing MongoDB as a Windows service means you do not have to manually start mongod every time.

Step 3 — Start, stop, or restart the MongoDB service

Do this in: Windows Services app
  1. Open the Start menu and search for Services
  2. Open the Services application
  3. Find the service named MongoDB
  4. Right-click it and choose Start, Stop, or Restart
  • Use this when the server is not running
  • Use restart after configuration changes or troubleshooting

Step 4 — Optional manual start, only if the service setup was skipped or failed

Run in: Command Prompt or PowerShell as Administrator
▶ OPTIONAL MANUAL SERVER START
cd C:\
md \data\db
"C:\Program Files\MongoDB\Server\<version>\bin\mongod.exe" --dbpath="C:\data\db"
Use this only if needed. Most students should use the Windows service installed by the MSI wizard instead.
If this works, the terminal usually shows a message similar to “waiting for connections”.

Step 5 — Install the MongoDB shell

  • Go to the MongoDB Shell download page
  • Download the Windows 64-bit MSI installer for mongosh
  • Run the installer
  • If the installer offers it, allow mongosh.exe to be added to your PATH
This step uses the browser + installer wizard. You are still not typing MongoDB commands yet.

Step 6 — Verify mongosh and connect to your local server

Run in: Command Prompt or PowerShell
▶ VERIFY + CONNECT
mongosh --version
mongosh
  • mongosh --version checks that the shell installed correctly
  • mongosh connects to your local MongoDB server if it is running
If mongosh is “not recognized”, close and reopen the terminal. If it still fails, add the mongosh install folder to your PATH.
